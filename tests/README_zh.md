# 测试规范

## 目的

这个目录用于存放 `picho` 的聚焦型自动化测试。

在新增测试之前，先确认它是否在保护有意义的行为：
- 新增的公共组件
- 行为变更
- 回归修复
- 配置解析与校验
- 工具或 runner 的边界场景

仅文档改动通常不需要测试。

## 运行测试

统一只使用 `uv` 命令：

```bash
uv run pytest
```

如果当前环境里还没有可导入的项目包，导致测试导入失败，先使用 editable mode 安装仓库本身：

```bash
uv pip install -e .
```

运行单个文件：

```bash
uv run pytest tests/test_read_tool.py
```

运行单个测试：

```bash
uv run pytest tests/test_read_tool.py -k csv
```

## 文件命名

- 测试文件使用 `test_<subject>.py`
- 测试函数使用 `test_<behavior>()`
- 当有助于可读性时，把相关行为放在同一个文件里

## 测试风格

- 优先写小而确定性的测试
- 在合理情况下，每个测试只覆盖一条行为断言路径
- 使用 Arrange / Act / Assert 结构
- 使用清晰的 fixture 搭建，避免隐藏耦合
- 注释尽量少；如确实需要，统一使用英文

## 隔离规则

- 不要依赖外部网络访问
- 不要依赖真实模型 provider 或真实凭据
- 不要读取 `.env` 或其他密钥文件
- 不要依赖临时目录之外的机器特定路径
- 优先使用 `tmp_path`、`monkeypatch` 和轻量 fake，而不是真实服务

## 何时补测试

在以下情况下应新增或更新测试：
- 引入新的组件或模块
- Bug 修复改变了可观察行为
- 新增配置分支或扩展点
- 工具协议、输出形态或错误路径发生变化

如果对一个非平凡改动选择不补测试，需要说明原因。

## 提交前检查

在结束改动前，确认：
- 测试名能清晰描述行为
- 断言对应用户可观察到的行为
- 测试在不同机器上保持稳定
- 测试可以通过 `uv run pytest` 运行
- 测试或辅助脚本没有依赖 `sys.path` 注入这种头部写法
